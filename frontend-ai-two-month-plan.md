# Web 前端工程师 AI 时代两个月提升计划

## 计划目标

计划周期：8 周。

建议投入：

- 工作日：每天 1.5 小时。
- 周末：每天 3-6 小时。

两个月后应完成 3 类成果：

1. 一个可演示的 AI 前端项目：支持流式输出、工具调用、错误处理、历史上下文。
2. 一份前端工程能力证明：性能优化报告、测试用例、组件重构记录。
3. 一套面试或晋升表达材料：项目 README、技术文章、简历项目描述。

## 每日固定节奏

| 时间 | 内容 |
| --- | --- |
| 20 分钟 | 阅读官方文档、源码文章或技术文章 |
| 50 分钟 | 写代码实践 |
| 20 分钟 | 复盘今天学到的内容和遇到的问题 |
| 10 分钟 | 用 AI 辅助总结、生成测试、检查代码 |

## 第 1 周：能力盘点与前端基本功

本周目标：明确当前能力短板，重新建立前端基础知识地图。

| 日期 | 行动细节 | 产出 |
| --- | --- | --- |
| 周一 | 建一个能力表，按 TypeScript、框架、工程化、性能、测试、Node、AI 应用分别给自己打 1-5 分；每项写清楚“我会什么 / 不会什么 / 想补什么”。 | 个人能力雷达 |
| 周二 | 练 TypeScript：泛型、联合类型、类型收窄、Pick、Omit、Partial、Record；每个类型写一个真实业务例子，比如用户、订单、消息。 | 10 个 TS 练习 |
| 周三 | 复习组件设计：把一个页面拆成 page、feature、component、hook、service；重点思考数据从哪里来、状态放哪里。 | 组件拆分图 |
| 周四 | 学习浏览器渲染流程：输入 URL、HTML 解析、CSSOM、layout、paint、composite；用自己的话写一遍。 | 渲染流程笔记 |
| 周五 | 找一个你做过的页面，列出 3 个问题：代码结构、性能、可维护性、交互体验。 | 改造清单 |
| 周六 | 新建项目：Vite + TypeScript + React/Vue，配置 ESLint、Prettier、路径别名、基础目录。 | 可运行项目 |
| 周日 | 写项目 README，说明技术栈、目录结构、启动命令、开发规范。 | 工程模板 v1 |

### 本周验收标准

- 完成一张能力雷达图。
- 完成一个可运行的前端项目模板。
- 写清楚当前最需要补的 3 个能力短板。

## 第 2 周：前端工程质量

本周目标：建立可维护、可测试、可持续迭代的工程习惯。

| 日期 | 行动细节 | 产出 |
| --- | --- | --- |
| 周一 | 设计目录结构：`pages` 放页面，`features` 放业务模块，`components` 放通用组件，`services` 放请求，`utils` 放纯函数。 | 目录规范 |
| 周二 | 写 Button、Modal、Input/Form 三个基础组件，支持 loading、disabled、error、size。 | 基础组件 |
| 周三 | 给组件写单元测试：测试点击、禁用、错误展示、边界输入。 | 至少 8 个单测 |
| 周四 | 加 Playwright，写登录、提交、列表筛选这类真实用户流程测试。 | 至少 2 条 E2E |
| 周五 | 配置 `typecheck`、`lint`、`test`、`build`，合成一个 `check` 命令。 | 质量脚本 |
| 周六 | 选择一个旧页面或 Demo 页面，按新目录重构，去掉重复逻辑。 | 重构代码 |
| 周日 | 写重构复盘：之前哪里乱，现在怎么拆，带来了什么收益。 | 重构说明 |

### 本周验收标准

- 项目具备基础组件、单元测试、E2E 测试。
- 可以通过一个命令完成类型检查、代码规范检查、测试和构建。
- 完成一份重构前后对比说明。

## 第 3 周：性能与用户体验

本周目标：掌握真实用户体验相关的性能分析和优化方法。

| 日期 | 行动细节 | 产出 |
| --- | --- | --- |
| 周一 | 学习 LCP、CLS、INP，分别写清楚它们衡量什么、常见问题、优化手段。 | 性能指标笔记 |
| 周二 | 用 Chrome Performance 分析页面，记录长任务、重复渲染、资源加载耗时。 | 性能问题截图 |
| 周三 | 优化首屏：路由懒加载、图片压缩、骨架屏、减少首包依赖。 | 首屏优化记录 |
| 周四 | 优化交互：列表虚拟滚动、防抖节流、memo/computed、减少无效 state。 | 交互优化记录 |
| 周五 | 补齐 loading、empty、error、retry、offline 状态，不让页面只有成功态。 | 状态组件 |
| 周六 | 做一次 Lighthouse 或手动性能对比，记录优化前后指标。 | 性能报告 |
| 周日 | 把性能优化写成面试表达：问题、分析、方案、结果。 | 面试素材 |

### 本周验收标准

- 能解释 LCP、CLS、INP 的含义和优化方式。
- 完成一次真实性能分析。
- 输出一份可用于面试的性能优化报告。

## 第 4 周：AI Chat 基础

本周目标：从普通前端开发进入 AI 应用前端开发。

| 日期 | 行动细节 | 产出 |
| --- | --- | --- |
| 周一 | 学习 LLM 基础：prompt、上下文、token、temperature、结构化输出；每个概念写一句业务解释。 | AI 基础笔记 |
| 周二 | 写聊天 UI：消息列表、输入框、发送按钮、用户消息、助手消息。 | Chat 页面 |
| 周三 | 接入流式输出，做到边生成边展示；增加停止生成按钮。 | Streaming Demo |
| 周四 | 设计消息状态：pending、streaming、done、error，并分别展示不同 UI。 | 消息状态机 |
| 周五 | 加 Markdown、代码块复制、重新生成、清空会话。 | 完整聊天体验 |
| 周六 | 把请求逻辑抽成 `useChat` 或 service，避免 UI 和请求混在一起。 | 可复用 Hook |
| 周日 | 写 README：功能截图、核心技术、流式输出实现方式。 | AI Chat Demo |

### 本周验收标准

- 完成一个基础 AI Chat 页面。
- 支持流式输出、停止生成、错误展示、重新生成。
- 能说清楚 AI Chat 前端和普通接口请求的差异。

建议两个月围绕一个主项目持续迭代，而不是每天换一个小 Demo。

| 模块 | 功能 |
| --- | --- |
| 聊天区 | 流式输出、停止生成、重新生成 |
| 文档区 | 文档列表、选中文档、引用片段 |
| 工具区 | 模拟查询订单、生成摘要、导出结果 |
| 可靠性 | 错误重试、超时提示、答案置信度 |
| 工程化 | TypeScript、测试、性能优化、README |

### 简历描述示例

```text
独立设计并实现 AI 知识库工作台，支持流式响应、工具调用、引用溯源、错误重试与 E2E 测试；针对长文本渲染和交互响应进行性能优化，沉淀可复用的 AI 对话组件和业务工具调用流程。
```

## 执行原则

1. 每天必须留下一个小产出，不要只看资料。
2. 每周必须有一个可展示成果。
3. 优先看官方文档，再看教程和视频。
4. 所有 Demo 都要写 README。
5. 所有重要功能都要有 loading、empty、error、retry 状态。
6. 用 AI 提效，但关键代码必须自己 review。
7. 复盘时重点记录“我为什么这样设计”，而不仅是“我做了什么”。
